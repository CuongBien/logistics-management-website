## Scope

### In scope

- All **functional requirements** `FR-001` … `FR-015` from `spec.md`
- All **acceptance criteria** `AC-001` … `AC-080` (grouped by theme in test cases below)
- All **success criteria** `SC-001` … `SC-005`
- Five **user scenarios** (standard receive, partial + force close, rejection, idempotent ERP retry, QA disposition)
- Technical decisions in `plan.md`: DDD encapsulation, EF unique keys, soft link for `DispositionLog.InboundLineId`, PostgreSQL + EF Core

### Out of scope

- Load / performance testing (not specified in spec)
- Full ERP contract certification (integration adapters marked non-goal in spec)
- Browser / Next.js UI (non-goal)
- Production Keycloak / multi-tenant soak tests

## Test strategy

| Level | Purpose | Framework | Target count (initial) |
|-------|---------|-----------|-------------------------|
| Unit | Domain invariants: quantities, force close, line/header status, `InventoryItem.ChangeStatus` | xUnit + FluentAssertions (optional) | ~18 |
| Integration | EF model: unique indexes, no FK on soft link, allocation sums | xUnit + EF InMemory or Testcontainers PG | ~8 |
| Application | MediatR handlers: duplicate receipt, force close publish, receive validation | xUnit + Moq | ~6 |
| E2E / API smoke | HTTP: create → receive → force close; auth headers | xUnit + `WebApplicationFactory` (if added) or manual Postman | ~4 |
| Manual | Overage escalation, SLA approval (open items) | Checklist in spec Open Items | ~2 |

**Acceptance threshold:** All **P1** cases must pass in CI before merge. **P2** failures need documented waiver. **P3** optional.

## Test cases by user scenario

### Scenario 1: Standard full receive

**Automation:** Unit + API smoke  
**Pass criteria:** Header `Completed`, all lines `Completed`, no `ShortageQty`, no rejections.

### Scenario 2: Partial receive + force close

**Pass criteria:** Line stays `PartiallyReceived` until command; after force close `ShortageQty` set; header completes with exceptions if rejects present.

### Scenario 3: Receive with rejection

**Pass criteria:** Rejection reason required; putaway only to compatible bin type (when FR-010 enforced).

### Scenario 4: Idempotent ERP retry

**Pass criteria:** Second create with same idempotency key fails; consumer replay does not duplicate (see `ShipmentSortedConsumer`).

### Scenario 5: QA disposition

**Pass criteria:** Status change via `InventoryItem.ChangeStatus`; log written after success; soft link resolvable.

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| InMemory provider does not enforce all PG unique constraints | False confidence | Add dedicated PG integration suite for TC-FR001-01 |
| `SourceRef` string vs Guid parsing for SC-005 | Event not published in edge cases | Explicit tests for Guid and non-Guid `SourceRef` |
| Open overage policy | Tests contradict future FR-005 change | Gate on `/speckit-clarify` outcome; version spec |
| Flaky MassTransit publish mocks | Intermittent CI | Strict mock ordering or `Verify` after `SaveChanges` only |
